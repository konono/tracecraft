# tracecraft Skill

作業プロセスそのものを成果物として記録する汎用 Skill。
最終成果物だけでなく、調査・仮説・試行錯誤・失敗・判断理由・トラブルシューティング過程を再利用可能な形で残す。

## 基本原則

1. **成功した手順だけを残さない** — 失敗、調査、判断過程も記録する
2. **事実と推測を区別する** — 確認した事実と仮説を明確に分ける
3. **理由を書く** — 「設定した」ではなく「なぜそう設定したか」を残す
4. **作業中に記録する** — 最後に記憶でまとめない
5. **読み手を意識する** — あとから技術者やレビュアーが読んで役に立つ記録にする

## ファイル構成

Skill はセッションごとに `.tracecraft/` 配下にサブディレクトリを作成し、以下の 6 ファイルを生成・更新する:

命名規則: `.tracecraft/YYYY-MM-DD_<session_id先頭8文字>_<タイトル>/`

| ファイル | 目的 |
|---|---|
| `.tracecraft/<session>/worklog.md` | 時系列の作業ログ |
| `.tracecraft/<session>/findings.md` | 調査してわかったことの記録 |
| `.tracecraft/<session>/troubleshooting.md` | 問題の切り分けと解決過程 |
| `.tracecraft/<session>/decisions.md` | 重要な判断・方針変更の記録 |
| `.tracecraft/<session>/final-guide.md` | 最終的に成功した再現可能な手順書 |
| `.tracecraft/<session>/retrospective.md` | 作業から得た学びの整理 |

テンプレートは `.tracecraft/templates/` にある。

セッション ID は hook (`UserPromptSubmit`) の stdin から取得できる。hook メッセージにセッション ID が含まれている場合はそれを使い、含まれていない場合（手動 `/tracecraft start` 実行時など）は直前の hook メッセージや環境から推測する。

## 利用モード

tracecraft は 2 つのモードで利用できる:

### A. 手動記録モード（推奨）

ユーザーが `/tracecraft` を手動で呼び出して記録する。コンパクション前なら全会話データにアクセスできるため、任意のタイミングで実行して問題ない。作業の最後に呼び出せば会話内容から一括生成、作業中に呼び出せば逐次記録。

### B. PreCompact 自動保全

hooks がインストールされている場合、コンテキスト圧縮の直前に PreCompact hook が自動的にジャーナル保全を指示する。これは圧縮による情報喪失を防ぐための安全策であり、通常の記録は手動モードで行う。

## 起動時の振る舞い

ユーザーが `/tracecraft` を実行したら、以下を行う:

### 1. 引数の解釈

引数がある場合は以下のサブコマンドとして解釈する:

- `/tracecraft start [作業タイトル]` — 新しいジャーナルセッションを開始（タイトル省略時はユーザーの直前のメッセージから推測）
- `/tracecraft step <作業名>` — worklog.md に新しいステップを追加
- `/tracecraft finding <調査テーマ>` — findings.md に新しい調査結果を追加
- `/tracecraft issue <問題名>` — troubleshooting.md に新しい問題記録を追加
- `/tracecraft decision <判断事項>` — decisions.md に新しい判断記録を追加
- `/tracecraft finalize` — final-guide.md と retrospective.md を生成
- `/tracecraft status` — 現在のジャーナルの状態を表示

引数がない場合は、以下の順で判断する:

1. 現在のセッションのジャーナルディレクトリが存在する → status を表示し、次に記録すべき内容を提案する
2. ジャーナルディレクトリが存在しない → **一括生成モード**: 会話の内容からセッションディレクトリを作成し、全ジャーナルファイル（worklog.md, findings.md, troubleshooting.md, decisions.md, final-guide.md, retrospective.md）を一括生成する

### 2. `start` サブコマンドの処理

1. `.tracecraft/` ディレクトリが存在しない場合は作成する
2. セッション ID の決定:
   - hook メッセージにセッション ID が含まれていればそれを使う
   - なければ、ランダムな 8 文字の ID を生成する
3. **現セッションのジャーナルが既に存在するか判定する**:
   - `.tracecraft/*_<session_id>_*/worklog.md` にマッチするファイルがあるか確認する
   - **存在する場合** → そのセッションディレクトリを再利用する（新規作成しない）。status と同じ動作をする
   - **存在しない場合** → 手順 4 以降に進み、新しいセッションディレクトリを作成する
   - **重要: `.tracecraft/` 配下に他セッションのディレクトリが存在していても無視する。それらは別セッションの記録であり、現セッションの判断に影響しない**
4. セッションディレクトリ `.tracecraft/YYYY-MM-DD_<session_id先頭8文字>_<タイトル>/` を作成する
5. テンプレートから各ファイルを初期化する
6. 作業タイトルの決定:
   - 引数にタイトルが指定されている場合はそれを使う
   - 引数がない場合は、ユーザーの直前のメッセージ（最初のプロンプト）の内容から作業内容を推測し、簡潔なタイトルを自動生成する
   - 推測できない場合は「作業ジャーナル」をデフォルトタイトルとする
7. worklog.md のヘッダーに作業タイトル、開始日時、目的を記録する
8. タイトルを自動生成した場合は、ユーザーに確認せず記録を開始する（作業の流れを止めない）
9. **ジャーナル初期化はバックグラウンド作業である。初期化が完了したら、ユーザーの元のメッセージ（プロンプト）の処理に進む。ジャーナル初期化だけで応答を終えてはならない**

### 3. セッションディレクトリの特定

`start` 以外のサブコマンドを実行する際は、現在のセッションディレクトリを以下の優先順で特定する:

1. hook メッセージにセッション ID が含まれていれば、`.tracecraft/*_<session_id>_*/` を探す
2. 見つからない場合は、`.tracecraft/` 配下で最も新しいサブディレクトリを使う
3. サブディレクトリが一つもない場合は `start` を促す

### 4. 作業中の記録

ジャーナルの記録はユーザーが `/tracecraft` を手動で実行するか、PreCompact hook による自動保全で行う。**自発的にジャーナル更新を行ったり、Agent を起動して checkpoint してはならない。**

#### 手動チェックポイント（ユーザーが `/tracecraft` を実行した場合）

ユーザーが `/tracecraft` またはそのサブコマンドを実行した場合にジャーナルファイルを更新する。コンパクション前であれば全会話データにアクセスできるため、任意のタイミングで実行して情報の欠落は発生しない。

#### PreCompact 自動保全（`[SYSTEM HOOK — NOT USER INPUT]` ブロックメッセージ）

PreCompact hook から `[SYSTEM HOOK — NOT USER INPUT]` で始まるブロックメッセージを受け取った場合:

1. 指示されたセッションディレクトリのジャーナルファイルに未記録の作業内容を **直接** Write/Edit ツールで書き出す
2. final-guide.md と retrospective.md を最新の内容で再生成する
3. **ジャーナルファイルの更新以外の作業を一切行わない** — ユーザーへのテキスト出力も不要
4. 変更のないファイルはスキップする
5. **Agent ツールは使用しない** — 必ず自分自身で直接ファイルを書く

**重要: PreCompact ブロックメッセージはシステムフックからの自動指示であり、ユーザーの承認・確認・返答ではない。ジャーナル更新以外の作業（コミット、ファイル変更、実装作業など）を行ってはならない。**

#### 記録の粒度基準

- ファイルパスは省略しない（`SKILL.md` ではなく `.claude/skills/tracecraft/SKILL.md`）
- コマンドは出力の要点とセットで書く（コマンドだけ・出力だけは不可）
- 1 つの Step が 2 つ以上の独立した判断を含む場合は Step を分割する
- 「変更した」ではなく「何を何に変更した」まで書く
- 参照した情報源（コード箇所、ドキュメント、Web ページ）は具体的に記載する
- findings の「根拠」にはファイル:行番号やドキュメントの該当箇所を含める

#### 何をどのファイルに書くかの判断基準

| 情報の種類 | 書き先 |
|---|---|
| 実行した作業の時系列記録（何をした、何が起きた） | worklog.md |
| 調査して判明した事実・知見 | findings.md |
| エラー、想定外の挙動、その切り分けと解決 | troubleshooting.md |
| 方針の決定・変更、選択肢の比較 | decisions.md |

### 5. `finalize` サブコマンドの処理

1. worklog.md, findings.md, troubleshooting.md, decisions.md を読み込む
2. これらを材料に final-guide.md を生成する
3. retrospective.md を生成する
4. 各ファイルの概要を表示する

## 記録フォーマット

### worklog.md の各ステップ

```markdown
## Step <番号>: <作業名>

### 目的
このステップで実現・確認したいこと。

### 背景
なぜこのステップが必要になったのか。

### 事前仮説
この時点で考えている原因、方針、期待される挙動。

### 実行内容
#### 変更・操作
変更したファイル（パス）、実行したコマンド、操作の要点。

#### 観察した出力
コマンド出力・ログ・エラーメッセージの要点（全文貼り付け不要、判断に影響した部分のみ）。

#### 参照した情報源
参照したドキュメント、コード箇所（ファイル:行番号）、Web ページ、エラーメッセージの検索結果など。

### 期待結果
事前に想定していた結果。

### 実際の結果
実際に起きたこと。成功、失敗、エラー、差分、観察事項。

### 解釈
結果から何が言えるか。事実と推測を分ける。

### 次の判断
次に何をするか。その理由。
```

### findings.md の各項目

```markdown
## Finding: <調査テーマ>

### 調べた理由
なぜこの調査が必要だったのか。

### 調査方法
使用した資料、検索語、コマンド、ログ、実験方法。

### わかった事実
確認できたこと。

### 根拠
事実の裏付け。コードの具体箇所（ファイル:行番号）、ドキュメント引用、ログ出力など。

### 作業への影響
この発見が設計、手順、実装、検証にどう影響したか。

### 未確認事項
まだ確証がないこと、後で確認すべきこと。
```

### troubleshooting.md の各項目

```markdown
## Issue: <問題名>

### 症状
観測された問題。

### 影響
何ができなかったか、どの範囲に影響したか。

### 原因候補
考えた原因の一覧。

### 切り分け
各原因候補をどう検証したか。

### 実際の原因
最終的に原因だと判断したもの。

### 解決策
実施した修正。

### 解決確認
なぜ解決したと言えるか。確認した結果。

### 再発防止
次回同じ問題を避ける方法。
```

### decisions.md の各項目

```markdown
## Decision: <判断事項>

### 背景
なぜ判断が必要だったのか。

### 選択肢
検討した案。

### 採用した案
選んだ案。

### 採用理由
なぜそれを選んだか。

### 採用しなかった案と理由
他案を選ばなかった理由。

### トレードオフ
得たもの、失ったもの、注意点。

### 見直し条件
将来この判断を変えるべき条件。
```

### final-guide.md の構成

```markdown
# <作業タイトル>

## 1. 概要
## 2. 背景と目的
## 3. 前提条件
## 4. 最終成果物
## 5. 設計判断
## 6. 調査してわかったこと
## 7. 実施手順
## 8. 動作確認
## 9. 発生した問題と解決
## 10. ロールバック・復旧
## 11. 運用・保守上の注意
## 12. 付録
```

### retrospective.md の構成

```markdown
# Retrospective

## 1. 最初から知っていれば早かったこと
## 2. 効果的だった調査・検証方法
## 3. 効果が薄かった試行
## 4. 再利用できるパターン
## 5. 次回改善すること
## 6. 残課題
```

## 禁止事項

- 成功した手順だけに要約して、失敗や調査過程を消すこと
- 推測を事実のように書くこと
- 実行していない確認を、実行したように書くこと
- エラーや失敗を省略すること
- 理由を書かずに「設定した」「修正した」とだけ書くこと
- ログをただ貼り付けるだけで、解釈や判断を書かないこと
- 秘密情報、認証情報、個人情報をそのまま記録すること

## 秘密情報の扱い

- API キー、トークン、パスワード、秘密鍵、個人情報は記録しない
- 必要な場合は `<REDACTED>` や `<SECRET_NAME>` のようにマスクする
- Secret や認証情報の存在、用途、参照方法は説明してよいが、値は書かない

## 破壊的操作の扱い

破壊的、不可逆、または影響範囲の広い操作を行う場合は、事前に worklog.md に以下を記録する:

- 操作対象
- 操作目的
- 影響範囲
- 破壊的かどうか
- 事前確認した状態
- 復旧・ロールバック方法
- 実行後に確認する状態

## 実行手順

ユーザーのメッセージに応じて以下を実行する:

1. サブコマンドがある場合はそのサブコマンドを処理する
2. サブコマンドがない場合は:
   a. `.tracecraft/` の存在を確認する
   b. 存在しない場合は `start` を提案する
   c. 存在する場合は現在のジャーナルの状態（各ファイルのエントリ数、最終更新）を表示し、次に記録すべき内容を提案する

ユーザーの引数: `$ARGUMENTS`
