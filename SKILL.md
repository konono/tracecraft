# tracecraft Skill

作業中の調査・仮説・試行錯誤・失敗・判断理由を構造化して記録し、再現可能な成果物・手順書・振り返りドキュメントを生成する汎用 Skill。

## なぜこの Skill が必要か

AI エージェントに作業を任せると、最終的に成功した手順や成果物は得られる一方で、以下が失われがちです:

- 途中で何を調べたのか、なぜ調べたのか
- どの仮説を立てたのか
- どの方法を試して失敗したのか
- なぜ最終的な方法を選んだのか

この Skill は **作業プロセスそのものを成果物として扱い**、人間があとからレビュー・再現・学習・監査できるようにします。

## 想定用途

- ソフトウェア開発、インフラ構築、デバッグ、技術調査
- PoC、検証作業、設計検討、ドキュメント作成
- 移行作業、運用手順の整備、自動化スクリプト作成
- モデルやサービスのデプロイ、API 連携、データ処理
- その他、試行錯誤を伴う作業全般

## ファイル構成

```
.claude/skills/tracecraft.md         # Skill 定義ファイル
SKILL.md                            # このファイル（Skill ドキュメント）
.tracecraft/
  templates/
    worklog.md                      # 時系列作業ログのテンプレート
    findings.md                     # 調査結果のテンプレート
    troubleshooting.md              # トラブルシューティングのテンプレート
    decisions.md                    # 判断記録のテンプレート
    final-guide.md                  # 再現手順書のテンプレート
    retrospective.md                # 振り返りのテンプレート
```

Skill 使用時に `.tracecraft/` 配下に実際の記録ファイルが生成されます。

## 使い方

### サブコマンド

| コマンド | 説明 |
|---|---|
| `/tracecraft start [作業タイトル]` | 新しいジャーナルセッションを開始(タイトル省略可) |
| `/tracecraft step <作業名>` | worklog.md に新しいステップを追加 |
| `/tracecraft finding <調査テーマ>` | findings.md に調査結果を追加 |
| `/tracecraft issue <問題名>` | troubleshooting.md に問題記録を追加 |
| `/tracecraft decision <判断事項>` | decisions.md に判断記録を追加 |
| `/tracecraft finalize` | final-guide.md と retrospective.md を生成 |
| `/tracecraft status` | 現在のジャーナル状態を表示 |
| `/tracecraft` (引数なし) | 状態確認と次の記録の提案 |

### 典型的なワークフロー

```
1. /tracecraft start Kubernetes クラスタの本番移行
2. (作業を進めながら、適宜 step / finding / issue / decision を記録)
3. /tracecraft finalize
```

### 作業中の自動記録

Skill をアクティブにした状態で作業すると、以下のタイミングで自動的に記録が更新されます:

- **worklog.md**: 新しい作業ステップの開始時、コマンド実行前後、想定外の結果が出たとき
- **findings.md**: ドキュメント・ソースコード・ログの調査時、新事実の判明時
- **troubleshooting.md**: エラー発生時、問題の切り分け時、問題解決時
- **decisions.md**: 方針の決定・変更時、複数選択肢からの選択時

## 各ファイルの役割

### worklog.md — 時系列作業ログ

作業の全ステップを時系列で記録します。各ステップには以下を含みます:

- 目的、背景、事前仮説
- 実行内容（コマンド、ファイル、ツール）
- 期待結果と実際の結果
- 解釈と次の判断

### findings.md — 調査結果

調査してわかったことを記録します:

- 調べた理由、調査方法
- 確認した事実
- 作業への影響
- 未確認事項

### troubleshooting.md — トラブルシューティング

問題の発生から解決までの過程を記録します:

- 症状、影響範囲、発生条件
- 原因候補と切り分け
- 実際の原因、解決策
- 解決確認、再発防止策

### decisions.md — 判断記録

重要な判断や方針変更を記録します:

- 背景、選択肢
- 採用した案と理由
- 採用しなかった案と却下理由
- トレードオフ、見直し条件

### final-guide.md — 再現可能な手順書

作業完了後に生成する、再現可能な手順書です:

- 概要、前提条件、最終成果物
- 設計判断、重要な発見
- 実施手順、動作確認
- ロールバック手順、運用上の注意

### retrospective.md — 振り返り

作業から得た学びを整理します:

- 最初から知っていれば早かったこと
- 効果的だったアプローチ
- 効果が薄かった試行
- 再利用できるパターン

## 基本方針

### 記録の原則

1. **最終的に成功した手順だけを残さない** — 作業中に調べたこと、なぜ調べたか、何がわかったかを記録する
2. **仮説・検証・結果・判断を分けて記録する** — 構造化された記録にする
3. **失敗した試行も記録する** — エラー、原因候補、切り分け、解決策を残す
4. **設計判断を記録する** — 選択肢、採用理由、却下理由、トレードオフを残す
5. **事実と推測を区別する** — 確認した事実と仮説を明確に分ける
6. **作業の途中で記録する** — 最後に記憶だけでまとめない
7. **読み手を意識する** — あとから技術者やレビュアーが読んで役に立つ記録にする

### 禁止事項

- 成功した手順だけに要約して、失敗や調査過程を消すこと
- 推測を事実のように書くこと
- 実行していない確認を、実行したように書くこと
- エラーや失敗を省略すること
- 理由を書かずに「設定した」「修正した」とだけ書くこと
- ログをただ貼り付けるだけで、解釈や判断を書かないこと

### 秘密情報の扱い

- API キー、トークン、パスワード、秘密鍵、個人情報は記録しない
- 必要な場合は `<REDACTED>` や `<SECRET_NAME>` のようにマスクする
- Secret の存在、用途、参照方法は説明してよいが、値は書かない

### 破壊的操作の扱い

破壊的・不可逆・影響範囲の広い操作の前に、worklog.md に以下を記録する:

- 操作対象、操作目的、影響範囲
- 破壊的かどうか
- 事前確認した状態
- 復旧・ロールバック方法
- 実行後に確認する状態


## 記録例

### worklog.md の記録例

```markdown
## Step 1: 既存の認証フローの調査

### 目的
現在の JWT ベースの認証フローの全体像を把握し、移行対象のコンポーネントを特定する。

### 背景
OAuth2 への移行計画を立てるために、まず現状を正確に理解する必要がある。

### 事前仮説
認証処理は auth-service に集約されており、他のサービスは JWT の検証のみ行っていると想定。

### 実行内容
- `grep -r "jwt" src/` で JWT 関連のコードを検索
- `src/auth/` 配下のコードを読み、認証フローを追跡
- API ゲートウェイの設定ファイルを確認

### 期待結果
JWT の生成・検証が auth-service に集約されていること。

### 実際の結果
JWT の生成は auth-service だが、検証ロジックが 3 つのサービスに分散していた:
- auth-service: トークン生成、リフレッシュ
- api-gateway: リクエストごとの検証
- user-service: プロフィール更新時の独自検証

### 解釈
**事実**: 検証ロジックが 3 箇所に分散している。
**推測**: user-service の独自検証は過去のバグ修正で追加された可能性がある（git blame で確認予定）。

### 次の判断
user-service の独自検証の経緯を git blame で調査する。移行時にこの分散を解消するか、維持するかの判断材料にする。
```

### troubleshooting.md の記録例

```markdown
## Issue: トークンリフレッシュ時の 401 エラー

### 症状
OAuth2 プロバイダからのトークンリフレッシュが 401 Unauthorized で失敗する。
リフレッシュトークン自体は有効期限内。

### 影響
ユーザーセッションが 1 時間で切れ、再ログインが必要になる。

### 原因候補
1. リフレッシュトークンのスコープ不足
2. クライアント認証の方式が間違っている（client_secret_post vs client_secret_basic）
3. リフレッシュトークンが rotation 設定でワンタイムになっている

### 切り分け
1. **スコープ**: トークンのペイロードを確認 → offline_access スコープが含まれている → 除外
2. **クライアント認証方式**: プロバイダのドキュメントを確認 → `client_secret_basic` が必要だが、コードは `client_secret_post` で送信していた → **該当**
3. **Rotation**: 未検証（原因 2 で解決したため）

### 実際の原因
クライアント認証方式の不一致。コードが `client_secret_post`（body に含める）で送信していたが、
プロバイダは `client_secret_basic`（Authorization ヘッダー）を要求していた。

### 解決策
`src/auth/oauth2-client.ts` の `refreshToken()` メソッドで、
認証方式を `client_secret_basic` に変更。

### 解決確認
- リフレッシュトークンのリクエストが 200 OK を返すことを確認
- 新しいアクセストークンでリソースサーバーにアクセスできることを確認
- 1 時間以上のセッション維持を確認

### 再発防止
OAuth2 プロバイダ連携時は、プロバイダのメタデータエンドポイント
（`.well-known/openid-configuration`）から `token_endpoint_auth_methods_supported` を
確認してからクライアントを実装する。
```

## ヒント

- **作業の粒度**: 1 ステップは「1 つの意図を持った操作」程度。細かすぎると読みにくく、粗すぎると情報が失われる
- **記録のタイミング**: 操作の前に意図を、操作の後に結果を書く。まとめて後から書くと記憶が曖昧になる
- **失敗の価値**: 失敗した試行は、同じ問題に遭遇した人にとって最も価値がある情報
- **finalize は材料から生成**: final-guide.md は worklog/findings/troubleshooting/decisions の内容を元に生成する。記録が充実していれば、質の高い手順書が生成できる
