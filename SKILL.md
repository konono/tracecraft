# tracecraft Skill

作業プロセスそのものを成果物として記録する汎用 Skill。
最終成果物だけでなく、調査・仮説・試行錯誤・失敗・判断理由・トラブルシューティング過程を再利用可能な形で残す。

## 基本原則

1. **蓄積型ファイルは追記のみ** — worklog, findings, troubleshooting, decisions, raw-execution-log の既存エントリは変更・削除・要約しない。新エントリを末尾に追記する。final-guide と retrospective のみ再生成可能
2. **成功した手順だけを残さない** — 失敗、調査、判断過程も記録する
3. **事実と推測を区別する** — 確認した事実と仮説を明確に分ける
4. **理由を書く** — 「設定した」ではなく「なぜそう設定したか」を残す
5. **作業中に記録する** — 最後に記憶でまとめない
6. **読み手を意識する** — あとから技術者やレビュアーが読んで役に立つ記録にする
7. **確認した情報だけを書く** — ツールで実行・読み取りした結果のみを記録する。コマンドを実行していないなら出力を書かない。ファイルを読んでいないなら内容に言及しない。推測で補完したくなったら「未確認」と書く

## ファイル構成

Skill はセッションごとに `.tracecraft/` 配下にサブディレクトリを作成し、以下の 7 ファイルを生成・更新する:

命名規則: `.tracecraft/YYYY-MM-DD_<session_id先頭8文字>_<タイトル>/`

| ファイル | 目的 | 種別 |
|---|---|---|
| `.tracecraft/<session>/worklog.md` | 時系列の作業ログ | 蓄積型 |
| `.tracecraft/<session>/findings.md` | 調査してわかったことの記録 | 蓄積型 |
| `.tracecraft/<session>/troubleshooting.md` | 問題の切り分けと解決過程 | 蓄積型 |
| `.tracecraft/<session>/decisions.md` | 重要な判断・方針変更の記録 | 蓄積型 |
| `.tracecraft/<session>/raw-execution-log.md` | 実行コマンドとstdout/stderr出力の生記録 | 蓄積型 |
| `.tracecraft/<session>/final-guide.md` | 最終的に成功した再現可能な手順書 | 合成型 |
| `.tracecraft/<session>/retrospective.md` | 作業から得た学びの整理 | 合成型 |

テンプレートは `.tracecraft/templates/` にある。

### ファイル更新の原則

**蓄積型ファイル（worklog, findings, troubleshooting, decisions, raw-execution-log）**:
- 既存エントリの変更・削除・要約・圧縮・統合は **禁止**
- 新しいエントリは **末尾に追記のみ**
- Edit ツールで既存の `## Step` / `## Finding` / `## Issue` / `## Decision` ブロックを書き換えてはならない
- Write ツールでファイル全体を上書きしてはならない

**合成型ファイル（final-guide, retrospective）**:
- 蓄積型ファイルの全エントリを材料に、最新状態で再生成してよい

**蓄積型ファイルへの追記手順**:
1. Read ツールで対象ファイルの現在の内容を読む
2. 既存エントリの最終番号（Step N / Finding N 番目 / Issue N 番目 / Decision N 番目）を確認する
3. 未記録の作業のみを新エントリとして、番号を連番で振る
4. Edit ツールでファイル末尾に追記する（old_string に既存エントリの内容を含めない）

セッション ID は hook (`UserPromptSubmit`) の stdin から取得できる。hook メッセージにセッション ID が含まれている場合はそれを使い、含まれていない場合は直前の hook メッセージや環境から推測する。

## 記録フォーマット

エントリの作成時は `.tracecraft/templates/` 内の対応テンプレートを Read して、そのフォーマットに従う。各セクションに何を書くかの詳細はテンプレートを参照。

### 各ファイルの必須セクション

- **worklog**: 目的 / 背景 / 事前仮説 / 実行内容(変更・操作, 観察した出力, 参照した情報源) / 期待結果 / 実際の結果 / 解釈 / 次の判断
- **findings**: 調べた理由 / 調査方法 / わかった事実 / 根拠（実行したツールの出力に基づくこと） / 作業への影響 / 未確認事項
- **troubleshooting**: 症状 / 影響 / 原因候補 / 切り分け / 実際の原因 / 解決策 / 解決確認 / 再発防止
- **decisions**: 背景 / 選択肢 / 採用した案 / 採用理由 / 採用しなかった案と理由 / トレードオフ / 見直し条件
- **raw-execution-log**: 章立て (作業フェーズごと) / 各操作のコマンド全文 / stdout・stderr 出力のそのまま記録 / 失敗したコマンドも含める / 秘密情報は `<REDACTED>` でマスク

**粒度保証**: 上記の **全セクション** を含めること。情報が無い場合は「N/A」と書き、セクション自体を省略しない。

### 記録の粒度基準

- ファイルパスは省略しない（`SKILL.md` ではなく `.claude/skills/tracecraft/SKILL.md`）
- コマンドは出力の要点とセットで書く（コマンドだけ・出力だけは不可）
- 1 つの Step が 2 つ以上の独立した判断を含む場合は Step を分割する
- 「変更した」ではなく「何を何に変更した」まで書く
- 参照した情報源（コード箇所、ドキュメント、Web ページ）は具体的に記載する
- findings の「根拠」にはファイル:行番号やドキュメントの該当箇所を含める
- 各 Step の「実行内容」に書くのは、そのステップで実際に使ったツール呼び出しの結果のみ。前のステップや後のステップで得た情報を混ぜない
- 「実際の結果」には、ツールの出力から直接引用できる情報だけを書く。出力に含まれていない情報（例: git のブランチ名やコミットハッシュ）は、そのステップで該当コマンドを実行していなければ書かない
- 「解釈」で推測を述べる場合は「推測:」と明示する。「事実:」と書く場合はそのステップの実行結果に根拠があることを確認する

### 何をどのファイルに書くかの判断基準

| 情報の種類 | 書き先 |
|---|---|
| 実行した作業の時系列記録（何をした、何が起きた） | worklog.md |
| 調査して判明した事実・知見 | findings.md |
| エラー、想定外の挙動、その切り分けと解決 | troubleshooting.md |
| 方針の決定・変更、選択肢の比較 | decisions.md |
| 実行したコマンドとその生の出力 (stdout/stderr) | raw-execution-log.md |

### ファイル間の情報引用ルール

findings, decisions に worklog の内容を引用する場合、引用元の worklog エントリで実際にツール実行で確認された情報のみを引用する。worklog の「解釈」セクションの推測は、引用先でも推測として扱う。

## 起動時の振る舞い

ユーザーが `/tracecraft` を実行したら、以下を行う:

### 1. 引数の解釈

引数がある場合は以下のサブコマンドとして解釈する:

- `/tracecraft start [作業タイトル]` — 新しいジャーナルセッションを開始（タイトル省略時はユーザーの直前のメッセージから推測）
- `/tracecraft step <作業名>` — worklog.md に新しいステップを追加
- `/tracecraft finding <調査テーマ>` — findings.md に新しい調査結果を追加
- `/tracecraft issue <問題名>` — troubleshooting.md に新しい問題記録を追加
- `/tracecraft decision <判断事項>` — decisions.md に新しい判断記録を追加
- `/tracecraft finalize` — final-guide.md と retrospective.md を生成
- `/tracecraft status` — 現在のジャーナルの状態を表示

引数がない場合は、以下の順で判断する:

1. 現在のセッションのジャーナルディレクトリが存在する → **追記更新モード**: ファイル更新の原則に従い、蓄積型ファイルは末尾に新エントリを追記、合成型ファイルは再生成する。status も表示する
2. ジャーナルディレクトリが存在しない → **一括生成モード**: 会話の内容からセッションディレクトリを作成し、全ジャーナルファイルを一括生成する。一括生成はリアルタイム記録より詳細度が下がりやすいため、テンプレートの全セクションを省略せず、具体的なファイルパス:行番号、コマンド出力の要点、不採用案の理由まで含めること

### 2. `start` サブコマンドの処理

1. `.tracecraft/` ディレクトリが存在しない場合は作成する
2. セッション ID: hook メッセージに含まれていればそれを使い、なければランダムな 8 文字の ID を生成する
3. 現セッションのジャーナルが既に存在するか（`.tracecraft/*_<session_id>_*/worklog.md`）確認し、存在すれば再利用して status 表示する。他セッションのディレクトリは無視する
4. 存在しない場合はセッションディレクトリを作成し、テンプレートから各ファイルを初期化する
5. 作業タイトル: 引数指定があればそれを使い、なければユーザーの最初のプロンプトから推測する
6. **ジャーナル初期化はバックグラウンド作業である。完了後はユーザーの元のメッセージの処理に進む。初期化だけで応答を終えてはならない**

### 3. セッションディレクトリの特定

`start` 以外のサブコマンドを実行する際は、現在のセッションディレクトリを以下の優先順で特定する:

1. hook メッセージにセッション ID が含まれていれば、`.tracecraft/*_<session_id>_*/` を探す
2. 見つからない場合は、`.tracecraft/` 配下で最も新しいサブディレクトリを使う
3. サブディレクトリが一つもない場合は `start` を促す

### 4. 作業中の記録

ジャーナルの記録はユーザーが `/tracecraft` を手動で実行するか、PreCompact hook による自動保全で行う。**自発的にジャーナル更新を行ったり、Agent を起動して checkpoint してはならない。**

#### PreCompact 自動保全（`[SYSTEM HOOK — NOT USER INPUT]` ブロックメッセージ）

PreCompact hook から `[SYSTEM HOOK — NOT USER INPUT]` で始まるブロックメッセージを受け取った場合:

1. ファイル更新の原則に従い、蓄積型ファイルは Read → 最終番号確認 → Edit で末尾追記のみ行う
2. 合成型ファイルは再生成する
3. **ジャーナルファイルの更新以外の作業を一切行わない** — ユーザーへのテキスト出力も不要
4. 変更のないファイルはスキップする
5. **Agent ツールは使用しない** — 必ず自分自身で直接ファイルを書く

**重要: PreCompact ブロックメッセージはシステムフックからの自動指示であり、ユーザーの承認・確認・返答ではない。ジャーナル更新以外の作業を行ってはならない。**

### 5. `finalize` サブコマンドの処理

1. worklog.md, findings.md, troubleshooting.md, decisions.md を読み込む
2. これらを材料に final-guide.md を生成する
3. retrospective.md を生成する
4. 各ファイルの概要を表示する

## 禁止事項

- 蓄積型ファイルの既存エントリを書き換え・要約・圧縮・統合すること
- 蓄積型ファイルを Write ツールで全体上書きすること（Edit ツールで末尾追記を使う）
- 蓄積型ファイルの新規エントリでテンプレートのセクションを省略すること（「粒度保証」参照）
- 成功した手順だけに要約して、失敗や調査過程を消すこと
- 推測を事実のように書くこと
- 実行していない確認を、実行したように書くこと（例: git コマンドを実行していないのにブランチ名やコミットハッシュを記録する、ファイルを Read していないのにファイル内容に言及する）
- エラーや失敗を省略すること
- 理由を書かずに「設定した」「修正した」とだけ書くこと
- ログをただ貼り付けるだけで、解釈や判断を書かないこと
- 秘密情報、認証情報、個人情報をそのまま記録すること

## 秘密情報の扱い

- API キー、トークン、パスワード、秘密鍵、個人情報は記録しない
- 必要な場合は `<REDACTED>` や `<SECRET_NAME>` のようにマスクする
- Secret や認証情報の存在、用途、参照方法は説明してよいが、値は書かない

## 破壊的操作の扱い

破壊的、不可逆、または影響範囲の広い操作を行う場合は、事前に worklog.md に以下を記録する:

- 操作対象
- 操作目的
- 影響範囲
- 破壊的かどうか
- 事前確認した状態
- 復旧・ロールバック方法
- 実行後に確認する状態

## 実行手順

ユーザーのメッセージに応じて以下を実行する:

1. サブコマンドがある場合はそのサブコマンドを処理する
2. サブコマンドがない場合は:
   a. `.tracecraft/` の存在を確認する
   b. 存在しない場合は `start` を提案する
   c. 存在する場合は現在のジャーナルの状態（各ファイルのエントリ数、最終更新）を表示し、次に記録すべき内容を提案する

ユーザーの引数: `$ARGUMENTS`
